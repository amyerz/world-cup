# World Cup 2026 — Meta Portal App

A sideloadable Android app for **Meta Portal touch and TV devices** (Portal, Portal+, Portal
Mini, Portal Go, Portal TV) that presents the World Cup 2026 dashboard across four tabs —
**Home** (schedule + live results), **Groups** (standings + FIFA ranking), **Bracket** (the
knockout tree), and **Countries** (team browser) — in a full-screen WebView. All assets
(React, fonts, flags, a results snapshot) are bundled locally, so the app launches and works
**fully offline**; when online it refreshes real 2026 data from a free, public-domain source.

The Portal app is the `android/` Gradle project plus the `web/` build pipeline that generates
its bundled web assets. (`design_handoff_worldcup_portal/` is the upstream design source and is
not part of the shipped app.)

## Quick Start with AI Tooling

The easiest way to build, deploy, and debug on a Portal is with **Claude Code** and the
bundled `portal` skill, which drives [`hzdb`](https://www.npmjs.com/package/@meta-quest/hzdb)
(Horizon Debug Bridge):

```
"Build the web bundle, install the APK on my Portal, and launch it."
```

The skill knows the Portal constraints (no Google Mobile Services, PNG-only launcher icon in
`mipmap-xxxhdpi/`, `MAIN`+`LAUNCHER` intent filter, API 28–29) and the build/deploy loop below.

## Getting Started

### Requirements
- **JDK 17** and the **Android SDK** (compileSdk 36, build-tools 36)
- **Node 20+** (for the esbuild-based web bundler)
- **`hzdb`** — `npm i -g @meta-quest/hzdb`
- A **Portal connected via USB-C** with **ADB enabled** (Settings → Debug → ADB)

### Build & deploy
```bash
# 1. Build the offline web bundle into android/app/src/main/assets/www/
cd web && node build.mjs            # add --force to re-download vendored React/fonts/flags

# 2. Build the APK
cd ../android && JAVA_HOME="$(/usr/libexec/java_home -v 17)" ./gradlew assembleDebug

# 3. Install + launch on the Portal
hzdb app install -r app/build/outputs/apk/debug/app-debug.apk
hzdb app launch com.erz.worldcup
```

### Debugging
```bash
hzdb adb logcat -d | grep WCWeb          # WebView console.* surfaces under tag "WCWeb"
hzdb capture screenshot -o /tmp/wc.png   # grab the current screen
```


## Architecture

This is a **WebView-hybrid** app, not a native one. A thin Java `Activity`
([MainActivity.java](android/app/src/main/java/com/erz/worldcup/MainActivity.java)) hosts a
full-screen `WebView` that loads a web app bundled inside the APK and runs entirely offline.
The UI is React (from the design handoff), compiled and vendored at build time — there is no
native Android UI, and no network is required at launch.

The repo is a small **monorepo** with three roles, wired together by the build:

```
design_handoff_worldcup_portal/   ──┐  (React/JSX design source — the input)
                                    │
web/build.mjs  (Node + esbuild)  ───┼─►  JSX→JS (minified), vendor React/fonts,
                                    │     prefetch flags, emit offline snapshot
                                    ▼
android/app/src/main/assets/www/   ◄──  GENERATED bundle (do not hand-edit)
                                    │
android/  (Gradle)  ── ./gradlew ───►  app-debug.apk  ──►  WebView on the Portal
```

So `web/` is **build-time tooling on your dev machine**, not a separate app and not a glasses
"display" app (that one lives in its own repo, `amyerz/wc26-glasses`). Editing happens in the
design source; `assets/www/` is always regenerated by `web/build.mjs`.

**How this differs from Meta's [portal-samples](https://github.com/meta-quest/portal-samples).**
Those samples are native, single-module **Kotlin + Jetpack Compose** apps (with the Gradle
project at the repo root) that demonstrate device APIs like CameraX and audio recording. This
app instead wraps an existing web UI in a WebView and keeps the Gradle project under `android/`
so the `web/` pipeline and design source can sit alongside it. Both are valid Portal apps — a
Portal app is simply an Android app meeting the Portal constraints (API 28–29, no Google Mobile
Services, `MAIN`+`LAUNCHER`, PNG launcher icon in `mipmap-xxxhdpi/`), all of which the
`android/` module satisfies. The hybrid route fits here because the UI already existed as a web
app; it trades away native capabilities (Compose, Smart Camera SDK) that this dashboard does
not need.

## Data & Configuration

The runtime reads `window.WC_CONFIG.source` (from `assets/www/config.js`). The adapter in
`web/src-extra/live-data.js` repopulates the app's existing `MATCHES` / `GROUPS` / `SCORERS` /
`BRACKET` globals — no screen code changes — matching teams by name plus an alias table.

| `source` | Cost | What it gives | Notes |
|---|---|---|---|
| **`openfootball`** (default) | **Free, no key** | Real 2026 schedule, groups, results (daily); standings computed client-side | [openfootball/worldcup.json](https://github.com/openfootball/worldcup.json) via jsDelivr; the bundled `assets/www/data/worldcup-2026.json` snapshot is the offline fallback. No in-match live minute (updates daily). |
| `apifootball` | Paid (for 2026) | Same + real-time minute + assists | api-sports.io's free tier blocks season 2026. Set `source:"apifootball"` and a paid `apiKey`. Budget-guarded at 95 req/day; live poll pauses when the screen is hidden. |

**The app needs nothing paid to run.** `refreshHours` (default 3) controls how often the
openfootball JSON is re-pulled; last-known data is cached in `localStorage` and the bundled
snapshot covers full offline use.

**Hidden screens.** The build ships only screens with a free live source. The **Top Scorers /
Stats** screen is built but disabled (`HIDDEN_TABS = ["stats"]` in `web/build.mjs`), so the
openfootball top-scorer data is computed but not displayed — delete that entry and rebuild to
re-enable it. FIFA ranking is shown on the **Groups** tab.

To enable the optional paid source, copy `web/src-extra/config.example.js` to `web/config.js`,
set your key, and rebuild. **`web/config.js` is `.gitignore`d** and must never be committed.
Note that `build.mjs` copies it into the APK, so the key ships inside the app bundle — see
**Security & privacy** before distributing a build with a key set.

## Portal Design Requirements

- `minSdk 28` (covers Portal + Portal+), `targetSdk 29`; **no Google Mobile Services**.
- Launcher needs a `MAIN`+`LAUNCHER` intent filter and a **PNG** icon in `mipmap-xxxhdpi/`.
- The ~64dp top system-overlay strip is handled in CSS: a `--portal-top-inset` band plus a
  dark navy scrim (`.portal-topscrim`) keep the white system pills legible.
- Tap targets and font sizes are enlarged for tabletop / across-the-room viewing distance.

## Security & Privacy

- All remote traffic is HTTPS; the manifest sets `usesCleartextTraffic="false"`.
- The WebView exposes **no `addJavascriptInterface`** native bridge, and the bundled UI uses
  React text nodes (no `innerHTML` / `eval` / `dangerouslySetInnerHTML`), so remotely fetched
  JSON is rendered as escaped text.
- Cached data in `localStorage` (`wc26.*`) is non-sensitive public match data and favorites.
- **API keys:** never embed a third-party key you can't afford to expose — `assets/www/config.js`
  is readable by anyone who unzips the APK. Keep `source` on the default `openfootball` for
  distributed builds, and treat any key that has lived in `web/config.js` as compromised once a
  build has been shared. See the hardening checklist in the audit notes.
